# MVP设计

核心验证：

无需用户注意力的 Agent 调度策略能不能给用户带来更多意图外的价值
需要的组件有这些
一个 Context 的组织库 - 可以简化为一个 markdown 文本库
一个 yarno agent，可以根据 context 中的所有内容去挖掘潜在需求并且调度 agent
一个通信协议A2A 类似，让 yarno 可以和其他 agent 去交互
一套适用于 A2A 世界的用户登录系统和支付管理系统，以及一个 agent 的开发规范（要求可以让 AI 大量生成这样的 agent 以解决早期 agent 生态不足的问题）

实现了这些就是实现了 AIOS
一个野心，未来的 AI 不需要 UI，只需要我们这一个 UI
这样把所有引用重新做一遍，所有目前需要人类来操作 UI 的核心痛点，全用 AI 做一遍

解决了什么问题？
用户侧：
- 用户大多数时候没有意图，无法清晰表述需求，但是想用 AI 做一些事情。（跨越鸿沟 AI 入门软件）
- 用户在有需求的时候不知道哪个 AI 可以解决问题，可以更好的解决问题。（工具发现）
- 用户想使用某个 AI 工具，却需要重复输入大量上下文。（提高使用体验）

供给侧：
- 开发者开发 AI 需要搞一堆配置，用户管理，上下文管理，支付系统，但这些都与验证 MVP 无关，浪费时间造轮子
- 开发者不需要经历长期的 context 收集，而是在 day1 就可以利用 context 去解决用户问题，也无需担心未来用户没有足量 context 导致产品力不足的问题
- 开发者不擅长推广营销，不知道在哪找种子用户

虽然目标瞄准的是跨越鸿沟后的市场
但在一开始，目标对象就是创新者和早期使用者

他们的唯一诉求就是好玩新奇，bug 什么的都无所谓
所以帅就完了，这也是我的需求，帅

那么所以一开始
就做一款终端产品，都是可以的，因为一开始就是给开发者做的


用户可以干什么？
1. 创建自己的 context library
2. 离开，No More Prompt

然后这个 agent 就会异步调用各种 agent，去尝试给你挖掘需求满足你的需求。
然后定期给你反馈结果，你当然可以介入指引，给他一些反馈，就像告诉你的助理你不喜欢喝冰美式。

当然你也可以直接聊天（但是这个因为不是创新点，所以有没有必要第一版本，为止）
或者直接招标发布需求。过去的时代是 product hunt，现在这个时代是 agent hunt

这里存在一个大问题，就是没有优质 context，没有优质 agent 的情况下没法儿拓展走出第一步。
但是考虑到第一波受众，是早期采用者和开发者，理念好玩就行。

Context 要自己手动创建，给他们写个文档，告诉他们怎么样效果好就行。
没有多少优质 Agent，但是依旧可以默默给我打工，他们看得清楚局限在哪里，暴露出这个最大的问题，让他们去解决自己的问题。
这个问题就解决了。

最一开始，甚至不需要任何的 context，就有一些好玩的 agent 可以去调用，最好。
然后生态就能成长起来。

那么最终，MVP 的实现只需要以下模块

- Context 管理系统：该版本只需要一个 markdown 文件供 agent 读写就可以。
Context 管理系统管理的是用户最核心的 context，这个 context 会随着 yarno agent 与外部世界的交互而发生变化，context 记录的核心在于用户的偏好和经历。
通过用户与 yarno 交互的反馈可以获取到用户的偏好。前期通过用户主动撰写来获取用户的经历。

- Yarno agent：一个核心 agent，向内可以读写 context，向外可以调度开放世界的 agent。
一个核心 agent，负责调取 context，挖掘需求，缝合 agent 能力和用户的潜在需求。初代版本只需要定时执行任务，分析需求、调度执行任务、检查结果、反馈 or 继续迭代结果、记录交互。
Yarno 负责和所有野生 agent 进行交互。有点像一个总经理，大 boss 只有模糊的偏好和需求，总经理需要揣度大 boss 的需求，然后用有限资源去尽可能满足大 boss 的潜在需求。并且还要看大 boss 的脸色去迭代。
因为钱是大 boss 给的啊！哈哈哈。

- 一套野生 agent 的撰写框架，可以被 yarno 找到并且调度
主要规范了 agent 的实现方式，必须要有什么功能，支持什么接口，如何被 yarno 发现和调度，需要哪些 context 作为支持等。

- 一套 A2A 协议：定义 yarno 和野生 agent 的交互方式
一套高效的 yarno 和野生 agent 的交互方式，不是你一嘴我一嘴然后一直执行。
而是异步的，yarno 发布任务，提供资源，野生 agent 执行，保存结果，反馈给 yarno，yarno 接受到反馈继续来执行，直到完成。
这里要定义基础的通信方式。

- Agent 的支付系统：先用简单的公共账簿做记录即可
支付系统决定的是任务的资源分配。意味着 yarno 要可以付钱给完成任务的野生 agent，也可能需要 yarno 去驱动野生 agent 去给用户挣钱。
所以需要一个账本。这个账本在整个 yarns 的世界是通用的。

- Agent 的登录系统：agent 之间的握手机制
Yarno 在代表用户去解决某个潜在需求时，需要使用用户 context 的时候，就要与用户给用户打报告，提示用户要用哪些内容。然后用户授权，意思是大boss 放权，那么yarno agent 就可以去代理大 boss，拿着资源和钱去和野生 agent 做交互。
野生 agent 也通过身份认证来创建session，保证任务的记录在同一空间内，不同用户之间不会串流。这里取代的是传统互联网的 google 登录和验证等一些列为人类设计的身份系统。

- 一个 agent 市场
用于让 yarno 对接需求和解决方案。只需要最简单的一个 json 格式的文本文件就可以。未来 yarns 公司需要维护这个 agent 市场的质量，去做声誉系统和评价系统等等，让 yarno 也知道谁靠谱谁不靠谱。
